# ⚙ KeepGoing｜Logic & Data Specification v3.0

---

## 0. この仕様書の位置づけ

このドキュメントは **KeepGoing の内部ロジックおよびデータ仕様**を定義する。

- **① Core & Experience Specification** を最上位判断基準とする
- UI・文言・画面構成・演出は扱わない
- 「どう決まり、どう進み、何が残るか」のみを扱う
- 実装・テスト・調整の唯一の根拠とする

---

### 0.1 本書の責任範囲

| 書くこと | 書かないこと |
|----------|--------------|
| 時間ルール・自動進行 | UIデザイン・文言 |
| スキップ・置換の考え方 | 体験の意味づけ（→①） |
| 状態遷移・優先順位 | 将来やるかもしれない話（→④） |
| 保存対象・保存タイミング | |
| 保持・破棄・匿名性 | |

---

### 0.2 用語定義

**Core準拠**
- **体験**：KeepGoingを開いてから終えるまでの全体
- **流れ**：体験の中で自動的に進行するストレッチの連続
- **区切り**：流れの中で、止まってよいと明示されるポイント

**Logic固有**
- 用語の正本は `docs/terminology.md` を参照する。
- **種目**：1つの運動単位（ストレッチや体幹など）。
- **メニュー**：1つ以上の種目を順番に実行するまとまり。
- **おまかせメニュー**：ホーム画面で自動的に選定・開始される日々のルーティン
- **カスタム種目**：ユーザーが作成した種目（UI表示: じぶん種目）
- **カスタムメニュー**：ユーザーが作成したメニュー（UI表示: じぶんのメニュー）
- **公開種目**：公開されたカスタム種目（UI表示: みんなの種目）
- **公開メニュー**：公開されたカスタムメニュー（UI表示: みんなのメニュー）
- **今日**：朝3時〜翌朝3時までの期間
- **種別**：流れの内部分類（STRETCH / CORE）。ユーザーには提示しない

## 1. 基本原則（Core拘束条件）

以下は **① Core & Experience** から渡される不変条件であり、
本仕様書は必ずこれを満たす。

- 判断は原則1回のみ（終了時）
- 起動時・体験中に判断を要求しない
- スキップは常に許可され、失敗扱いしない
- スキップ後もロジックは必ず成立する
- 記録はユーザーに意識させない
- 数値競争・評価は禁止

---

### 1.1 自動進行・時間設計の前提（Core→Logic受け渡し）

- 体験は明示的に開始されない
- アプリ起動と同時に論理的に開始状態に入る
- 時間・回数は体験上の意味を持たない
- Logic上でも **管理対象にしすぎない**

※ Logic は「制御」ではなく **流れを成立させる役割**を担う

---

### 1.2 判断回数制約の適用

- アプリ起動直後や体験中には判断を求めない
- 大きな強制アンケートや確認ダイアログを出さない
- 終了は、**設定された目標時間（デフォルト10分）に到達した時点** で自然に行われる（判断不要で「おつかれさま」となる）

※ 中断したい時だけ、ユーザーが能動的に停止画面から×ボタンを押す

---

### 1.3 スキップ処理の思想的拘束

- スキップは失敗状態を生成しない
- スキップ回数を評価・保存しない
- 次の内容への **置換処理** として扱う
- 成功／失敗フラグを生成しない

---

### 1.4 育成進行ロジックの拘束条件

- 育成は操作によって進まない
- 条件達成による明示的進行を行わない
- 行動の結果として **非線形・揺らぎを含んで進行**する
- ユーザーが最適化・管理できない構造を必須とする

### 1.4.1 ふわふわ（育成キャラクター）の進行ロジック

- **ライフサイクル**: 誕生から28日間を1世代のサイクルとする
    - Day 1〜7 (1週目): たまご期
    - Day 8〜21 (2・3週目): 妖精期
    - Day 22〜28 (4週目): 成体期
    - Day 29〜: お別れ分岐（成体到達 or 未到達で異なるフロー）
- **成長・進化条件**:
    - 妖精への進化: 誕生からの経過日数内で、ストレッチを実施した日数（活動日数）が **3日以上** 必要
    - 成体への進化: 誕生からの活動日数が累計 **10日以上** 必要
    - 条件を満たさない場合は進化が保留され、前の段階にとどまる（頑張らないと育たない）
- **妖精期のサイズ変化 (揺らぎ)**:
    - 妖精期において、活動日数に応じて見た目のサイズが3段階で変化する（例: 5日以下=小、6〜8日=中、9日以上=大）
    - 運動すればするほど顕著に大きく成長する仕様を体現する
- **お別れ分岐ロジック** (Day 29+):
    - `calculateFuwafuwaStatus` はさよなら時にも実際の `stage` と `activeDays` を返す
    - **成体到達 (stage === 3)**: 「お部屋にいくよ！」→ `pastFuwafuwas` に `finalStage: 3` で保存 → お部屋に常駐
    - **成体未到達 (stage !== 3)**: 「バイバイまたね！」→ `pastFuwafuwas` に保存されるが、お部屋には表示されない（`finalStage !== 3` でフィルタ）
- **ユニークタイプ選択** (`pickNextFuwafuwaType`):
    - 未登場タイプが残っている間は、過去に登場済みのタイプ＋現在のタイプを除外してランダム選出
    - 全10種を制覇した場合は、全タイプからランダム（同一連続も許可）
    - タイプ数は将来的に拡張可能

---

### 1.5 データ保存・保持の思想

- 保存はユーザーに意識させない
- 行動を評価する形で保存しない
- 保存対象は「事実」と「状態」に限定する
- 未実行・中断をマイナス状態として保存しない

---

## 2. 流れの構造

### 2.1 流れの開始

- 流れはアプリ起動と同時に論理的に成立する
- UI表示・カウント開始の前後は問わない

---

### 2.2 流れの終了（完了）

- **目標時間の完走**：設定された目標時間（デフォルト10分）分の種目がすべて完了した時点。自然に「おつかれさま」画面を表示して完了する。
- **ユーザーによる中断**：画面をタップして停止し、「✕ボタン」を押して終了した時点。

※ 完了時も、途中で中断した時も、そこまでの記録は保存される。

---

### 2.3 区切りの時間定義

- **小さな区切り**：約5分（種目により変動。短い余白を挟むのみ）
- **大きな区切り**：約15分（長時間設定時のリフレッシュ用小休止）
- **セッション完了**：設定された時間（デフォルト10分）に到達した時点

---

### 2.4 設定時間による制御

- 「無限に続いて大きな区切りで止まる」のではなく、「**目標時間分のプレイリストを生成・消費し切る**」構造である
- メニュー時間（目標時間）はユーザーが設定から変更可能

---

## 3. 種別と進行原則

### 3.1 種別ラベル（内部用）

現時点では以下の種別を内部的に保持する。
ユーザーには提示しない。

- **STRETCH**（ストレッチ中心）
- **CORE**（体幹中心）

※ 将来の拡張については④を参照

---

### 3.2 自動進行

- 種目は自動で選定・進行される
- ユーザーによる進行制御は存在しない
- 停止・中断は常に許可される

---

## 4. 状態管理

### 4.1 再生位置の原則

- ストレッチの途中状態は保持しない
- 未完了セッションの再開時は常にセッション先頭から
- 経過秒数の保存・復元は行わない

---

### 4.2 セッション中の画面遷移

- StretchSession overlay が最前面に表示されるため、通常のタブ移動は発生しない
- セッションを終了 / 完了した時だけ、開始時に決まった戻り先へ戻る

---

### 4.3 バックグラウンド時の状態

- バックグラウンドへ → 再生停止
- フォアグラウンド復帰 → 現在のストレッチIDを保持、先頭から再開

---

### 4.4 履歴の保持

- 今日実施したストレッチの履歴を保持する
- 昨日以前の履歴は「戻る」対象から除外
- 保持するのは順序とストレッチIDのみ

---

### 4.5 遡り制限

- 戻れる範囲は「今日」内に限定
- 日付境界（朝3時）を超えた履歴へは遷移不可

---

## 5. 種目選定ロジック

### 5.1 候補抽出条件

## 11. クラウド同期・認証

### 11.1 概要

- Supabase をバックエンドとして利用する（任意：未ログインでもアプリは完全に使える）
- ローカルファーストの設計：全てのデータはまずローカル（localforage / Zustand persist）に保存し、ログイン時にクラウドへ同期する
- ユーザーは同期を意識しない（自動処理）

---

### 11.2 認証

| 方式 | 詳細 |
|------|------|
| 匿名認証 | supabase.auth.signInAnonymously（初回起動時に自動実行） |
| メール/パスワード | supabase.auth.signInWithPassword / signUp |
| Google OAuth | supabase.auth.signInWithOAuth({ provider: 'google' }) |
| 必須/任意 | 任意（未ログインでも全機能利用可） |

- 初回起動時に自動的に匿名認証を実行（ユーザーに意識させない）
- アカウント登録時は匿名アカウントから実アカウントへアップグレード（`updateUser` で同一UIDを維持）
- Google OAuth登録時は `linkIdentity` で匿名アカウントにGoogleアイデンティティをリンク（失敗時はフォールバックで通常OAuth）
- 新規メール登録時はメール確認リンクを送付。確認前はログイン不可
- ログアウト後は匿名アカウントを再作成し、ローカルデータを保持

---

### 11.3 同期対象テーブル

| テーブル名 | 対応ローカルデータ | 主キー |
|-----------|------------------|--------|
| `family_members` | UserProfileStore（家族メンバー） | uuid |
| `sessions` | SessionRecord（セッション履歴） | uuid |
| `custom_exercises` | CustomExercise（カスタム種目） | (id, account_id) |
| `menu_groups` | MenuGroup（カスタムメニューグループ） | (id, account_id) |
| `app_settings` | アプリ設定（音量・通知・suspended等） | account_id（1件のみ）|
| `challenges` | チャレンジ定義（管理者が作成、全ユーザー閲覧可） | uuid |
| `challenge_completions` | チャレンジ達成記録 | uuid |
| `public_menus` | 公開メニュー（全ユーザー閲覧可） | uuid |

---

### 11.4 同期フロー

**初回ログイン時（initialSync）**
1. ローカルの全家族メンバーを `family_members` へ upsert
2. ローカルの全セッション履歴を `sessions` へ upsert
3. カスタム種目を `custom_exercises` へ upsert
4. カスタムメニューグループ（isPreset=false のみ）を `menu_groups` へ upsert
5. アプリ設定を `app_settings` へ upsert
6. オフラインキューに残っている操作があれば処理

**通常操作時（ローカル書き込み後）**
- ローカルに保存後、即座にクラウドへ push（upsert / delete）
- オフライン or Supabase未設定の場合はオフラインキューに enqueue
- オンライン復帰時にキューを自動処理

**定期処理**
- ログイン中、1分ごとにキューを処理（processQueue）

---

### 11.5 オフラインキュー

- localforage の `sync_queue` ストアにエントリを保存
- エントリは `{ id, table, operation, payload, createdAt }` の構造
- 処理は createdAt 順（古い順）
- 1件でも失敗した場合は中断し、次回に再試行

---

### 11.6 セキュリティ（RLS）

- 全テーブルに Row Level Security を有効化
- 基本ポリシー：`auth.uid() = account_id` のレコードのみ操作可能
- クライアント側で account_id を偽装してもサーバー側で拒否される

**特殊アクセス関数**：
- `is_teacher()`: 先生メール判定。先生は `family_members`, `sessions`, `app_settings` を全アカウント閲覧可
- `is_developer()`: 開発者メール判定。開発者は全テーブルを全アカウント閲覧可

**テーブル別の公開ポリシー**：
- `challenges`: 全ユーザー閲覧可（`using(true)`）。作成は `is_teacher()` のみ
- `challenge_completions`: 自分の達成記録のみ操作可。先生/開発者は全件閲覧可
- `public_menus`: 全ユーザー閲覧可（`using(true)`）。作成/削除は自分のレコードのみ

**RPC関数（security definer）**：
- `increment_download_count(menu_id)`: 公開メニューのDLカウント+1
- `fetch_active_public_menus(sort_by, max_count)`: 休止アカウントを除外した公開メニュー一覧（`app_settings.suspended` を内部JOINしてフィルタ）
- `suspend_account(target_account_id, is_suspended)`: アカウント休止/解除（`is_developer()` チェック）
- `delete_account_data(target_account_id)`: アカウントの全データ削除（`is_developer()` チェック）

---

### 11.7 Pull（リストア）と競合解決

**リストア**: 新端末ログイン時にクラウドからデータを取得してローカルに復元する（実装済み）。

**競合検出（detectConflict）**：
ログイン時にクラウドとローカルの両方にデータが存在するかを判定し、3パターンに分岐する。

| パターン | 条件 | 処理 |
|---------|------|------|
| `no_conflict_pull` | クラウドにデータあり、ローカルにデータなし | クラウドからpull |
| `no_conflict_push` | ローカルにデータあり、クラウドにデータなし | ローカルからpush |
| `conflict` | 両方にデータあり | ユーザーに選択を求める |

**競合解決ダイアログ**：
- 「クラウドのデータを使う」→ クラウドからpull
- 「このデバイスのデータを使う」→ ローカルからpush
- 一度解決したアカウントは `SYNCED_ACCOUNT_KEY` で記録し、次回以降は競合ダイアログをスキップ

**Append-onlyデータのマージ**：
- セッション履歴、カスタム種目、カスタムメニューは競合解決後にマージ（重複排除付き）

## 12. チャレンジ機能

### 12.1 概要

チャレンジは `先生チャレンジ` と `じぶんチャレンジ` を持つ。
両者は `対象を達成した日` または `期間中の合計実施数` を同じ計算エンジンで扱い、違いは保存先・開始トリガー・報酬の強さに寄せる。

| goal_type | text | MVP では `active_day` を中心に使う |
| window_days | int | 7 / 14 / 30 など |
| required_days | int | 必要達成日数 |
| status | text | `active` / `completed` / `ended_manual` / `ended_expired` |
| started_at | timestamptz | 作成と同時に入る開始時刻 |
| effective_end_date | date | 実際の終了日 |
| reward_granted_at | timestamptz | 報酬付与済み時刻 |
| created_at | timestamptz | 作成日時 |
| updated_at | timestamptz | 更新日時 |

**`challenge_completions`テーブル**：

| カラム | 型 | 説明 |
|--------|------|------|
| id | uuid | 主キー |
| challenge_id | uuid | チャレンジID |
| account_id | uuid | 達成者のaccount_id |
| member_id | text | 達成したメンバーのID |
| completed_at | timestamptz | 達成日時 |

---

### 12.3 進捗計算ロジック

- 進捗計算は `calendar` / `rolling`、`total_count` / `active_day` を共通エンジンで扱う
- **calendar**: `start_date` 〜 `end_date` をそのまま有効期間に使う
- **rolling**: `joined_at` または `started_at` を基準に `effective_end_date` を決める
- **total_count**: 対象期間内の対象実施数を合計する。`daily_cap` があれば1日上限をかける
- **active_day**: 対象期間内で `対象を1回以上やった日` を 1 日として数える
- 日数型では、同じ日に複数回やっても 1 日として扱う
- 目標到達時は `status = completed` とし、必要に応じて `challenge_completions` に記録する

---

### 12.4 参加ロジック

- 先生チャレンジは `参加 = 開始`
- rolling 型の先生チャレンジでは、参加時に `challenge_enrollments` を作成し、`effective_start_date / effective_end_date` を固定する
- calendar 型の先生チャレンジでも参加状態は enrollment で持つ
- `joinedChallengeIds` は旧ローカル参加記録との互換レイヤとして扱い、参加日時を必要とする処理の正本にはしない
- みんなでモード時は全アクティブユーザーを一括参加
- クラスレベルフィルタ: ユーザーの `classLevel` が `challenges.class_levels` に含まれるもののみ表示
- じぶんチャレンジは `作成 = 開始`
- じぶんチャレンジは個人モードでのみ作成し、`personal_challenges.started_at` を開始時刻とする

---

### 12.5 達成報酬

- 先生チャレンジは `reward_kind` に応じて `ほし` または `ちびふわふわ / メダル` を付与する
- じぶんチャレンジの報酬は `ほし1こ` に固定する
- 二重付与防止のため、報酬付与後は `reward_granted_at` を記録する
- 達成時に `reward_fuwafuwa_type` 相当の報酬がある場合は `family_members.chibifuwas` 配列に追加する
- ちびふわふわはふわふわ（育成キャラクター）とは別の小さなコレクションアイテム

## 13. 公開メニュー

### 13.1 概要

ユーザーが作成したメニューグループを他のユーザーに公開・共有する仕組み。

---

### 13.2 テーブル定義

**`public_menus`テーブル**：

| カラム | 型 | 説明 |
|--------|------|------|
| id | uuid | 主キー |
| name | text | メニュー名 |
| emoji | text | メニューアイコン |
| description | text | メニューの説明 |
| exercise_ids | text[] | 含まれる種目IDリスト |
| author_name | text | 公開者の名前 |
| account_id | uuid | 公開者のaccount_id |
| download_count | int | ダウンロード数（初期値0） |
| created_at | timestamptz | 公開日時 |

---

### 13.3 公開・非公開ロジック

- メニュー作成時に公開トグルがオンの場合、`public_menus` にINSERT
- 既存メニューの公開/非公開はメニューカードのボタンで切替
- 非公開にする場合は `public_menus` から該当レコードをDELETE（自分のaccount_idのみ）

---

### 13.4 インポートロジック

- 他のユーザーの公開メニューを「かしてもらう」でローカルに保存
- ローカルでは `imported-{publicMenuId}` のIDで `menu_groups` に保存
- インポート時に `increment_download_count` RPCでDLカウントを+1

---

### 13.5 おすすめアルゴリズム

- **トレンド**: 1週間以内に作成された人気メニュー（DL数順）
- **最新**: 作成日時が新しいメニュー
- **人気**: DL数が多いメニュー
- 各カテゴリから1件ずつ選出し、重複排除（メニュー名+種目リストの一致で判定）
- 合計3件をホーム画面のおすすめ行に表示

---

### 13.6 休止アカウントフィルタ

- `app_settings.suspended = true` のアカウントが公開したメニューは一般ユーザーの画面に表示しない
- `fetch_active_public_menus` RPC が `security definer` で `app_settings` を内部JOINして実現
- RPC未デプロイ時はフォールバックとして直接クエリ（フィルタなし）を実行

---
