## 1. 継続時の選択肢拡張（未実装）

### 1.1 現状

- 大きな区切りで「続ける」→ 同じ構造（約15分）が繰り返される
- 種別（STRETCH / CORE）は内部概念としてのみ存在
- ユーザーには提示しない

### 1.2 将来の拡張案

「続ける」を押した際に、以下のような選択肢を提示する：

- 「続ける」→ 通常の流れ（現状と同じ）
- 「体幹中心で続ける」→ CORE種別の流れ
- 「ストレッチ中心で続ける」→ STRETCH種別の流れ

### 1.3 今は入れない理由

- 判断回数制約（終了時のみ1回）との整合性が未検討
- 選択肢を増やすことで「考えなくていい」原則が崩れる可能性
- まずはシンプルな構造で体験を検証したい

### 1.4 拡張時の注意点

- 選択肢は「判断」にカウントされるか？
- Core §7「判断回数の制約」との整合性を必ず検討すること
- 選択肢を出すなら、選ばなくても自然に進む設計が必要
  - 例：何もしなければデフォルト（通常の流れ）で継続

## 2. 育成（ふわふわ）機能 ✅ 実装済み

### 2.1 実装内容

- ホーム画面にふわふわキャラクターを常時表示
- たまご → 妖精 → 大人 の3段階成長（`calculateFuwafuwaStatus`で日数ベース判定）
- 成長時にマイルストーンモーダルで「やってきた！」「かえった！」「そだったね！」と通知
- ユーザーごとに独立した育成状態を保持
- 複数ユーザー時は「みんなで！」ページで並べて表示

### 2.2 ライフサイクルと「お別れ」分岐 ✅ 実装済み

- 28日サイクル終了時、ふわふわの最終ステージに応じてお別れが分岐する
  - **成体（stage 3）到達** → 「お部屋にいくよ！」の金色フェアウェルモーダル → 「お部屋」に常駐（いつでも会える）
  - **成体未到達（stage 1 or 2）** → 「バイバイまたね！」のピンクフェアウェルモーダル → お部屋には入らない
- `calculateFuwafuwaStatus` はさよなら時にも実際のステージと活動日数を返す（お別れ分岐判定に使用）

### 2.3 ユニークタイプ選択 ✅ 実装済み

- 次のふわふわは過去に来たことのないタイプからランダム選出（`pickNextFuwafuwaType`）
- 全10種を制覇した場合は、直前のタイプ以外からランダム（同じ子が連続しない）
- タイプ数は今後増やしていく想定

### 2.4 お部屋（ギャラリー）✅ 実装済み

- 記録ページのサブタブ「お部屋」に成体まで育ったふわふわのみ表示（`finalStage === 3` フィルタ）
- 成体未到達のふわふわはお部屋に表示されない
- 空状態: 「まだ お部屋にはだれもいません / ふわふわを成体まで育てると、ここに引っ越してくるよ。」

### 2.5 思想的根拠（Core §12 準拠）

- **比較しない**：他ユーザーとの優劣なし。成長速度の差は見せない
- **競争させない**：ランキング・スコア的表示なし
- **管理させない**：ユーザーが育成を操作する要素なし（自動で変化する）
- **失敗を生まない**：成体に届かなくても「またね」で終わる（罰ではなく出会いの偶然性）

### 2.6 今後の拡張検討

- レア・色違いの導入（「偶然性・出会い」として）
- 生体メッセージ（褒めない・評価しない・指示しない第三の声）
- ふわふわの種類数を段階的に増やす

## 3. チャレンジシステム ✅ 実装済み

### 3.1 概要

先生（teacher）が期間限定のチャレンジを作成し、生徒が日々のストレッチで達成すると「ちびふわ」（ミニふわふわ）を獲得できる仕組み。

### 3.2 実装内容

#### データモデル
- **Challenge**: `id`, `title`, `exerciseId`, `targetCount`, `startDate`, `endDate`, `createdBy`, `rewardFuwafuwaType`
- **ChallengeCompletion**: `challengeId`, `accountId`, `memberId`, `completedAt`
- **ChibifuwaRecord**: `id`, `type`（ふわふわタイプ番号 0-9）, `challengeTitle`, `earnedDate`
- ストレージ：Supabase（`challenges`, `challenge_completions` テーブル）
- ちびふわはユーザーの Zustand ストアに `chibifuwas: ChibifuwaRecord[]` として永続化

#### 先生側（TeacherDashboard）
- 「生徒」「チャレンジ」のタブ切替
- チャレンジ作成フォーム：タイトル、種目（ドロップダウン）、目標回数、開始日・終了日、ちびふわタイプ（0-9選択）
- チャレンジ一覧表示・削除

#### 生徒側（HomeScreen → ChallengeCard）
- ホーム画面に期間中のチャレンジをカード表示
- ローカルセッションデータから進捗を自動カウント（`countExerciseInPeriod`）
- プログレスバーで達成度を可視化
- 目標到達時に自動でチャレンジ完了を記録 + ちびふわを付与

### 3.3 思想的根拠

- **パッシブ体験を妨げない**（Core §2.2）：チャレンジは任意のアクティブ体験。参加しなくても通常のストレッチ体験は一切変わらない
- **判断を増やさない**（Core §7）：進捗は自動カウント、達成時も自動で完了処理。ユーザーが「完了ボタン」を押す必要なし
- **失敗を生まない**（Core §10.2）：期間が過ぎても未達のペナルティなし。「未達成による損失」は禁止事項
- **育成の延長**（Core §12）：ちびふわは「頑張ったから報われる」ではなく「関わったから変わった」の体現

### 3.4 今後の拡張検討

- ちびふわギャラリー（コレクション一覧表示）
- チャレンジの自動提案（先生なしでも季節チャレンジなど）
- チャレンジ達成時のアニメーション演出強化

## 4. メニュー共有（みんなのメニュー） ✅ 実装済み

### 4.1 概要

ユーザーが作成したカスタムメニュー（くみあわせ）を他のユーザーと共有できる機能。公開・閲覧・もらう（インポート）の一連のフローを提供する。

### 4.2 実装内容

#### データモデル
- **PublicMenu**: `id`, `name`, `emoji`, `description`, `exerciseIds`, `authorName`, `accountId`, `downloadCount`, `createdAt`
- ストレージ：Supabase（`public_menus` テーブル）
- ダウンロードカウントの更新は Supabase RPC（`increment_download_count`）でアトミックに実行

#### 公開（メニューページ → GroupCard）
- 自分のカスタムメニューの各カードに「公開」ボタン（アカウント連携済みの場合のみ表示）
- ワンタップで Supabase に公開

#### 閲覧（PublicMenuBrowser）
- ボトムシートモーダルとして表示
- 人気順（ダウンロード数順）でメニュー一覧を表示
- 各カードに名前・作成者・ダウンロード数・種目プレビューを表示

#### もらう（importMenu）
- 「もらう」ボタンでローカルにカスタムメニューとして保存
- 同時にダウンロードカウントをインクリメント
- インポート済みのメニューは「もらい済み」と表示

#### アクセスポイント
- **メニューページ**：「みんなのメニュー」セクション → 「みんなのメニューを見る」ボタン
- **ホーム画面**：PopularMenusRow（人気メニュー横スクロール） → 「もっと見る」で全一覧

### 4.3 思想的根拠

- **アクティブ体験**（Core §2.2）：メニュー共有は慣れたユーザー向け。パッシブ体験（考えずに始まる）を妨げない
- **判断を強制しない**（Core §7）：ホーム画面の PopularMenusRow は提案のみ。タップしなければ自動で無視される
- **比較・ランキングの回避**（Core §10.1）：ダウンロード数はソート用で控えめに表示。ランキング画面は作らない

### 4.4 今後の拡張検討

- カテゴリ・タグでのフィルタリング
- メニューへのいいね・コメント
- 先生のおすすめメニュー機能

## 5. ホーム画面拡張 ✅ 実装済み

### 5.1 概要

ホーム画面をスクロール可能なレイアウトに変更し、ふわふわキャラクターに加えてチャレンジ、先生のメニュー、みんなのメニューを表示する。

### 5.2 実装内容

ホーム画面の構成（上から順）：

1. **PageHeader**（「ホーム」+ コンテキストバッジ）
2. **MagicTank**（今日の練習時間タンク）
3. **ふわふわキャラクター カルーセル**（スワイプ切替、ユーザー選択）
4. **ページネーションドット**
5. **チャレンジセクション**（`challenges.length > 0` のとき表示）
   - 「チャレンジ」見出し
   - ChallengeCard をリスト表示
6. **先生のメニューセクション**
   - `teacher_section` の先生メニューから、新着またはおすすめのものを 1-2 件表示
   - 先生カードは「くわしく見る」と「はじめる」を分け、ホームでの誤タップ開始を減らす
7. **みんなのメニューセクション**
   - トレンド/最新/人気から選んだ公開メニュー 3 件をカード表示
   - セクション内に `みんなの種目も見る` 導線を持ち、公開種目の発見をホーム内で継続する
   - 「もっと見る」で PublicMenuBrowser を開く

#### レイアウト変更
- `justifyContent: 'center'` を削除し、上詰めスクロールレイアウトに変更
- `overflowY: 'auto'` + `WebkitOverflowScrolling: 'touch'` でスムーズスクロール
- `paddingBottom: 90` で下部の「はじめる！」ボタンとの重なりを回避

### 5.3 思想的根拠

- **ホームは体験が始まっている状態**（Core §3.2）：ふわふわが最上位。チャレンジ・メニューは下部に配置し、スクロールしなければ見えない
- **判断を増やさない**（Core §7）：追加セクションは情報提示のみ。表示されるだけで操作を要求しない
- **パッシブ体験の保護**（Core §2.1）：「はじめる！」ボタンは固定フロート。追加セクションに埋もれない
- **つながりの可視化**：ホームでは `先生` と `みんな` を別セクションにしつつ、同じカードファミリーで見せることで関係性を明確にしながら画面の統一感を保つ

### 5.4 今後の拡張検討

- ちびふわギャラリーへの導線
- 週間・月間の振り返りカード
- 季節イベントバナー

## 6. Supabase テーブル構成（実装必要）

以下のテーブル・カラム・関数が新機能に必要：

### 6.1 テーブル

| テーブル名 | 用途 |
|---|---|
| `challenges` | チャレンジ定義（先生が作成） |
| `challenge_completions` | チャレンジ達成記録 |
| `public_menus` | 公開メニュー |

### 6.2 カラム追加

| テーブル | カラム | 型 |
|---|---|---|
| `family_members` | `chibifuwas` | `jsonb DEFAULT '[]'` |

### 6.3 RPC関数

| 関数名 | 用途 |
|---|---|
| `increment_download_count(menu_id uuid)` | ダウンロードカウントのアトミック加算 |

### 6.4 RLSポリシー

- `challenges`: 全員読み取り可、`is_teacher()` のみ作成・削除可
- `challenge_completions`: 自分の `account_id` のみ読み書き可
- `public_menus`: 全員読み取り可、自分の `account_id` のみ作成・削除可
